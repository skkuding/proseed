# 분석·계측 아키텍처

PROSEED의 사용자 행동 측정을 **프론트엔드는 GA4, 백엔드는 Observability(OTel/Prometheus)**
로 나눈 이유와 각 층이 답하는 질문을 정리한다.

## 배경

"사용자가 무엇을 하는지"를 알아야 제품을 개선할 수 있다. 그런데 측정 도구를 아무 데나
붙이면 안 된다. 도구마다 강점이 다르고, **필요한 정보가 어디에 존재하는지**가 다르기 때문이다.

## 결정

측정은 **진실이 있는 곳 + 질문이 자연스럽게 풀리는 곳**에 둔다. 그 결과 두 갈래로 나뉜다.

|             | 프론트엔드 → GA4                       | 백엔드 → Observability                      |
| ----------- | -------------------------------------- | ------------------------------------------- |
| 답하는 질문 | 여정·귀속 ("어디서 와서 어디서 이탈")  | 사실·정확성 ("정확히 몇 건, 추세")          |
| 진실의 위치 | 브라우저 (쿠키·referrer·UTM·세션·기기) | 서버 DB (실제 커밋된 상태)                  |
| 도구        | GA4 (`@next/third-parties`)            | OpenTelemetry 카운터 → Prometheus → Grafana |
| 구현        | PR #110                                | PR #111                                     |

### 프론트엔드 → GA4 인 이유

GA4의 가치는 행동을 **유입경로·세션에 묶는 것**이다. 그 맥락(어디서 왔나, 무슨 기기,
이전에 뭘 했나)은 전부 브라우저에만 존재한다.

- 백엔드에서 GA4로 보내려면(Measurement Protocol) 브라우저의 `client_id`가 필요한데
  서버엔 그 값이 없다 → 귀속·세션 연결이 깨진다.
- GA4는 마케팅/여정 도구다. "유입 채널별 가입 전환율", "등록까지 며칠" 같은 세션 질문에 맞다.
- 약점: 광고 차단으로 일부 이벤트가 유실된다. 하지만 여정은 전수가 아니라 "모양"만
  필요하므로 감수한다.

### 백엔드 → Observability 인 이유

서버는 **사실의 원천**이다. 프로젝트가 진짜 생겼는지(DB 커밋), 진짜 탈퇴했는지에 대해
광고 차단·위조·유실이 없다.

- "정확히 몇 건", "rate", "탈퇴 대비 가입", 운영 건강은 권위 있는 카운트가 필요한 질문이다.
- 백엔드는 이미 관측 스택(OTel → Prometheus → Grafana) 안에 있다. 비즈니스 카운터가
  인프라 지표와 같은 레일을 타므로 한곳에서 본다.
- 약점: per-user 여정·귀속이 불가능하고 저카디널리티 라벨만 담을 수 있다. 그 부분은
  GA4가 커버하므로 겹치지 않고 상호보완한다.

## 결과

- **같은 이벤트가 양쪽에 있어도 중복이 아니다.** 예: `project_created`는 GA4에서
  "유입→등록 전환율", Prometheus에서 "하루 몇 개 정확히" — 두 개의 렌즈다.
- **뒤집으면 둘 다 손해다.** 백엔드→GA4는 `client_id`가 없어 귀속을 잃고 Measurement
  Protocol과 씨름해야 한다. 프론트엔드→Observability는 정확성을 잃고(광고 차단·위조)
  수천 개 브라우저가 메트릭을 밀어 넣어 카디널리티·남용 문제가 생긴다.
- **현재 상태(net) 집계는 어느 쪽도 아닌 DB 쿼리 몫이다.** "지금 남아있는 회원 수"는
  Grafana의 PostgreSQL 데이터소스로 직접 조회한다.

## 남은 공백

per-user 코호트·리텐션 정밀 분석("월요일 가입자 중 7일 내 프로젝트 만든 %")은 GA4의
기본 기능으로도, 저카디널리티 메트릭으로도 깊게 답하기 어렵다. 필요가 커지면 이벤트 단위
제품 분석 도구(PostHog 등) 도입을 검토한다.
